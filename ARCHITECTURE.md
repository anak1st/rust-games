# 架构设计

## 当前目标

当前项目的目标不是直接实现多游戏框架，而是先用一个足够小的例子学习 `ratatui` 的基本组织方式。

当前实现就是一个最小的 `Counter App`：

- 使用 `ratatui` 绘制界面
- 使用 `crossterm` 读取键盘事件
- 在一个最小循环里完成“绘制 -> 处理输入 -> 更新状态”

这正是目前要做的内容，不再额外抽象成更大的游戏框架。

## 当前结构

当前源码结构如下：

```text
src/
  main.rs
  app.rs
```

说明：

- `main.rs` 负责程序入口和启动 `ratatui`
- `app.rs` 负责应用状态、事件处理和界面绘制

## 当前运行方式

程序启动后进入一个最小 TUI 界面。

当前交互规则：

- `Left`：计数减一
- `Right`：计数加一
- `q`：退出程序

## 当前数据流

当前程序的数据流非常简单：

```text
键盘事件 -> App 状态更新 -> ratatui 重绘
```

对应过程：

1. `main.rs` 创建 `App`
2. `main.rs` 通过 `ratatui::run` 启动终端环境
3. `app.rs` 内部进入循环
4. 每一轮先 `draw`
5. 再读取键盘事件
6. 根据按键更新 `counter` 或退出

## 模块职责

### `main.rs`

职责：

- 解析启动参数
- 创建 `App`
- 调用 `ratatui::run`
- 把终端交给 `App::run`

这一层保持尽量薄，不放具体界面逻辑。

### `app.rs`

职责：

- 保存应用状态
- 处理输入事件
- 决定何时退出
- 使用 `ratatui` 绘制界面

当前 `App` 内部主要包含两个状态：

- `counter`
- `exit`

## 当前实现重点

当前代码主要在学习下面几件事：

- 如何组织 `ratatui` 的主循环
- 如何把状态放进 `App`
- 如何响应 `crossterm` 键盘事件
- 如何实现 `Widget` 并把状态渲染出来
- 如何使用 `Block`、`Paragraph`、`Line`、`Text` 这些基础组件

这一步的重点是把 `ratatui` 的基本写法走通，而不是提前为未来做复杂抽象。

## 当前设计选择

当前刻意不做这些事情：

- 不拆成多模块事件系统
- 不引入 `Game trait`
- 不引入 `runtime.rs`
- 不引入多游戏注册表
- 不提前设计完整游戏架构

原因很简单：

- 现在更重要的是把 `ratatui` 的基本使用方式学明白
- 先把一个小程序写顺，比先搭大框架更适合 review

## 下一步方向

如果继续保持小步推进，后续更适合做的是：

1. 学 `Layout`
2. 学多个 widget 组合
3. 学输入和状态更新的拆分
4. 再决定是否进入“游戏框架”阶段

## 总结

当前项目的真实状态可以概括为：

```text
一个最小的 ratatui 计数器程序
```

它的目的不是展示完整架构，而是作为后续继续学习 `ratatui` 和逐步演进结构的起点。
