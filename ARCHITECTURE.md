# 架构设计

## 概述

这个项目的目标是用一套共享运行框架支持多个终端小游戏，例如贪吃蛇、俄罗斯方块等。

当前阶段的设计原则是尽量简单、够用：

- 一个统一的程序入口
- 一个共享的游戏主循环
- 一个 TUI 渲染文件
- 多个具体游戏实现

架构的重点不是做很重的抽象，而是让后续新增游戏时，不需要反复改启动流程和主循环。

## 设计目标

- 不同游戏复用同一套启动和运行流程
- 游戏逻辑与终端渲染分离
- 新增游戏时尽量只改游戏自身代码
- 第一版避免过度设计

## 当前范围

当前只考虑 TUI 渲染。

这意味着：

- 渲染代码集中在 `src/render.rs`
- 暂时不考虑多种渲染后端
- 暂时不为渲染做复杂抽象

当前真正需要守住的边界只有一条：

**游戏逻辑不能直接操作终端输出。**

## 核心流程

运行时的固定流程建议如下：

```text
输入 -> 游戏更新 -> 渲染数据 -> TUI 渲染
```

对应到主循环就是：

```text
1. 读取输入
2. 把输入交给当前游戏
3. 更新游戏状态
4. 让游戏生成当前帧的渲染数据
5. 渲染到终端
```

这条流程一旦稳定下来，后面无论增加多少游戏，主循环都不需要大改。

## 模块结构

第一版建议采用下面的结构：

```text
src/
  main.rs
  app.rs
  runtime.rs
  render.rs
  game/
    mod.rs
    snake.rs
    tetris.rs
```

## 模块职责

### `main.rs`

保持入口最小化。

职责：

- 解析启动参数
- 调用 `App::run()`

### `app.rs`

负责程序装配和启动。

职责：

- 选择运行哪个游戏
- 创建运行时
- 创建当前游戏实例
- 启动执行流程

这一层负责“把系统拼起来”，不负责具体游戏规则，也不负责终端绘制细节。

### `runtime.rs`

负责共享主循环。

职责：

- 读取用户输入
- 把输入转发给当前游戏
- 触发游戏更新
- 获取渲染数据
- 调用 TUI 渲染
- 控制退出时机

这一层应该只保存程序流程，不应该写任何具体游戏的规则。

### `render.rs`

负责终端渲染。

职责：

- 清屏
- 移动光标
- 输出字符或文本
- 绘制当前帧

这一层不应该包含任何游戏规则，例如：

- 碰撞判断
- 分数更新
- 方块下落
- 蛇的移动逻辑

### `game/`

负责存放具体游戏实现。

每个游戏的职责包括：

- 保存自己的状态
- 处理输入
- 更新规则
- 生成渲染数据
- 告知游戏是否结束

例如：

- `snake.rs`
- `tetris.rs`

## 多游戏的多态方式

为了支持多个游戏，多态应该放在游戏层。

推荐方式：

- 定义统一的 `Game` trait
- 每个游戏实现这个 trait
- 运行时通过 `Box<dyn Game>` 持有当前游戏

概念上可以理解为：

```rust
trait Game {
    fn handle_input(&mut self, input: Input);
    fn update(&mut self);
    fn render_data(&self) -> RenderData;
    fn is_over(&self) -> bool;
}
```

然后由不同游戏分别实现：

- `SnakeGame`
- `TetrisGame`

`runtime` 不需要知道当前具体是哪一个游戏，只需要通过统一接口调用即可。

这就是当前项目最合适的“多态”方式。

## 渲染数据

游戏代码不应该直接打印终端。

更合适的做法是：游戏把当前状态转换成一份简单的渲染数据，再交给 `render.rs`。

这份渲染数据可以很简单，例如：

- 字符网格
- 多行文本
- 带坐标的字符单元

这样分层之后：

- 游戏层决定“显示什么”
- 渲染层决定“怎么显示”

## 保持简单的部分

第一版不建议引入下面这些内容：

- 多渲染后端
- 复杂事件总线
- ECS
- 插件系统
- 很深的抽象层级
- 很多价值不高的小 trait

只要守住下面几个边界，这个架构就已经足够支撑项目继续长大：

- 游戏逻辑不直接渲染
- 渲染层不写游戏规则
- 主循环由 `runtime` 统一管理

## 后续扩展方式

后面新增一个游戏时，理想流程应该是：

1. 在 `src/game/` 下新增文件
2. 实现统一的 `Game` 接口
3. 在 `app.rs` 中接入该游戏
4. 复用同一套 `runtime`
5. 复用同一个 `render.rs`

目标不是完全不改老代码，而是尽量把改动限制在新游戏附近。

## 总结

当前阶段最合适的架构主线是：

```text
输入 -> 游戏状态 -> 渲染数据 -> TUI 渲染
```

对应到工程结构，就是：

- 一个共享 `runtime`
- 一个 TUI 渲染文件 `render.rs`
- 一个统一的游戏接口 `Game`
- 多个具体游戏实现

这套结构已经足够支持项目从单一游戏逐步扩展到多游戏，同时不会让架构变得过重。
