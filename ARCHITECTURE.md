# 架构设计

## 当前目标

当前项目的目标不是直接实现完整多游戏框架，而是先用一个足够小的例子学习 `ratatui` 的基本组织方式，并逐步过渡到多界面。

当前实现已经包含两个界面：

- 选择游戏界面
- 一个最小的“计数器”界面
- 一个带玩家蛇、AI 敌蛇和尸块转食物机制的“贪吃蛇”界面

当前重点是先把“界面切换 + 逻辑更新 + 渲染更新 + 输入处理 + 最小游戏抽象 + 状态弹窗”这条链路走通。

## 当前结构

当前源码结构如下：

```text
src/
  main.rs
  app.rs
  game/
    mod.rs
    counter.rs
    snake.rs
```

说明：

- `main.rs` 负责程序入口和启动 `ratatui`
- `app.rs` 负责应用状态、界面切换、暂停状态、事件处理和界面绘制
- `game/mod.rs` 定义最小 `Game` trait、`GameKind` 和共享类型
- `game/counter.rs` 实现最小的“计数器”游戏
- `game/snake.rs` 实现带敌蛇和基础 AI 的“贪吃蛇”游戏

## 当前运行方式

程序启动后进入一个最小 TUI 界面。

当前交互规则：

- 选择游戏界面
  - `Up/Down`：移动选中项
  - `Enter`：进入游戏
  - `q`：退出程序
- 计数器界面
  - `Left/Right`：修改计数
  - `Space`：切换暂停
  - `r`：由 `App` 重开当前游戏
  - `q`：返回选择游戏界面
  - `Esc`：返回选择游戏界面
- 贪吃蛇界面
  - `Arrows/WASD`：控制方向
  - `i`：在手动控制和 AI 控制之间切换
  - `Space`：切换暂停
  - `r`：由 `App` 重开当前游戏
  - `q`：返回选择游戏界面
  - `Esc`：返回选择游戏界面

当前还支持：

- 游戏进入特殊状态时显示居中弹窗
- 特殊状态弹窗会根据状态使用不同颜色
- 终端尺寸变化导致游戏内容区大小变化时，自动重开当前游戏
- 窗口过小时显示 `WindowTooSmall` 状态弹窗
- 游戏启动时把内容区大小传入游戏实例

启动方式：

- 默认进入主界面
- 可以通过 `--game counter` 直接进入计数器游戏
- 可以通过 `--game snake` 直接进入贪吃蛇游戏

## 当前数据流

当前程序的数据流非常简单：

```text
键盘事件 -> App / Game 状态更新 -> ratatui 重绘
```

对应过程：

1. `main.rs` 创建 `App`
2. `main.rs` 通过 `ratatui::run` 启动终端环境
3. `app.rs` 内部进入循环
4. 主循环先按固定间隔等待事件或 update tick
5. 输入事件到来时先更新 app 状态
6. 到达 update tick 时调用游戏逻辑更新
7. 状态发生变化后再 `render`

## 模块职责

### `main.rs`

职责：

- 解析启动参数
- 创建 `App`
- 调用 `ratatui::run`
- 把终端交给 `App::run`

这一层保持尽量薄，不放具体界面逻辑。

### `app.rs`

职责：

- 保存应用状态
- 处理输入事件
- 决定何时退出
- 管理当前界面和当前游戏
- 管理 app 层级的游戏状态
- 拆分逻辑更新和渲染更新
- 使用 `ratatui` 绘制界面

当前 `App` 内部主要包含这些状态：

- `exit`
- `screen`
- `game_index`
- `game`
- `game_size`
- `game_status`

当前 `app.rs` 的额外职责还包括：

- 用固定常量控制标题栏、底栏和状态栏尺寸
- 根据终端大小推导当前游戏内容区大小
- 在 resize 后判断是否需要重开当前游戏
- 统一渲染游戏状态弹窗
- 在窗口过小时阻止正常游戏更新并显示特殊状态
- 在准备、暂停、胜负和窗口过小时使用统一弹窗引导用户操作
- 通过固定 update tick 和独立 render 拆开主循环职责

### `game/mod.rs`

职责：

- 定义应用承载游戏所需的最小接口
- 定义可启动的 `GameKind`
- 让 `GameKind` 提供标题等类型级元数据
- 定义游戏尺寸 `GameSize`
- 定义 footer 使用的 `Instruction`

### `game/counter.rs`

职责：

- 实现当前的计数器游戏
- 管理计数器自己的内部状态
- 提供逻辑更新函数
- 提供内容区、状态区和帮助信息
- 在启动时接收内容区大小

### `game/snake.rs`

职责：

- 实现带玩家蛇和多条 AI 敌蛇的贪吃蛇
- 管理蛇身、方向、食物、尸块和分数
- 按固定帧数推进一次移动
- 让 AI 根据附近食物和风险位置自动转向
- 在敌蛇死亡后把尸块逐步转成可再次食用的食物
- 在玩家撞墙、撞到自己、其他蛇或尸块时切换为失败状态
- 在内容区过小时返回 `WindowTooSmall`

## 当前实现重点

当前代码主要在学习下面几件事：

- 如何组织 `ratatui` 的主循环
- 如何把状态放进 `App`
- 如何通过 trait 把游戏内容挂进 app 外壳
- 如何把游戏标题这类类型级元数据放到 `GameKind`
- 如何响应 `crossterm` 键盘事件
- 如何给逻辑更新增加固定频率
- 如何拆分 update 和 render 两类职责
- 如何根据不同界面切换不同的绘制内容
- 如何使用 `Block`、`Paragraph`、`Line`、`Text` 这些基础组件
- 如何使用 `Layout` 组织页面区域
- 如何在 app 层处理暂停、重开和弹窗
- 如何在 resize 后重建游戏实例

这一步的重点是把 `ratatui` 的基本写法走通，而不是提前为未来做复杂抽象。

## 当前设计选择

当前刻意不做这些事情：

- 不拆成多模块事件系统
- 不引入 `runtime.rs`
- 不引入复杂的多游戏注册表
- 不提前设计完整游戏架构

原因很简单：

- 现在更重要的是把 `ratatui` 的基本使用方式学明白
- 先把一个小程序写顺，比先搭大框架更适合 review

## 下一步方向

如果继续保持小步推进，后续更适合做的是：

1. 把不同界面的绘制继续拆小
2. 把 `GameStatus::Won` 接进实际逻辑
3. 继续用更多游戏验证 `Game` trait 是否合适
4. 再决定是否进入更完整的“游戏框架”阶段

## 总结

当前项目的真实状态可以概括为：

```text
一个带游戏选择页、固定外壳布局、状态弹窗，以及计数器和多蛇贪吃蛇两个游戏实现的 ratatui 程序
```

它的目的不是展示完整架构，而是作为后续继续学习 `ratatui` 和逐步演进结构的起点。
